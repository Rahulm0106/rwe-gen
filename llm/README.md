# RWE-Gen Pre-Execution Pipeline

Developer-facing pipeline for converting a clinician question into:

1. a validated study protocol JSON,
2. a concept-mapped OMOP protocol,
3. a populated OMOP SQL query template.

The pipeline **does not execute SQL**. Its scope ends after building the SQL that would be sent to an OMOP database.

## Overview

The system is organized into five Python modules:

- `cli.py` — developer CLI for running one stage or the full pipeline
- `api.py` — orchestration facade used by the CLI and future interfaces
- `llm_module.py` — Venice-backed protocol generation
- `concept_mapping_module.py` — concept mapping with remote IMO normalization plus local OMOP vocabulary matching
- `omop_sql_module.py` — fixed OMOP SQL template population only

For backward compatibility, the repository also includes `athena_module.py` as a thin shim that re-exports `concept_mapping_module.py`. The CLI also retains the legacy `athena` subcommand and `--athena-*` option names.

Pipeline flow:

```text
Clinician question
   -> LLM protocol generation
   -> Protocol schema validation
   -> Concept mapping
   -> SQL template population
```

## What the CLI can run

The CLI supports these commands:

- `run` — full pipeline: LLM -> concept mapping -> SQL
- `llm` — only protocol generation
- `concept-mapping` — only concept mapping on an existing protocol JSON file
- `athena` — legacy alias for `concept-mapping`
- `map` — short alias for `concept-mapping`
- `sql` — only SQL population on an existing mapped protocol JSON file
- `examples` — prints example questions and example calls

## Installation and setup

### Required files

You need:

- `protocol_schema_validator.json`
- the Python pipeline files
- local OMOP vocabulary files used by the concept mapping stage:
  - `CONCEPT.csv`
  - `CONCEPT_RELATIONSHIP.csv`
  - optional `CONCEPT_SYNONYM.csv`

### Python dependencies

Install the runtime dependencies used by the current codebase:

```bash
pip install requests jsonschema
```

### Venice API key

If you want to use the live LLM stage:

```bash
export VENICE_API_KEY="your_key_here"
```

### Remote IMO access through LOF

The remote concept-mapping path does **not** call public Athena. It uses your Leap of Faith proxy to call IMO Normalize. Export the LOF credentials used in your lab:

```bash
export client_id="your_lof_client_id"
export client_secret="your_lof_client_secret"
```

Accepted fallbacks for compatibility are also:

- `CLIENT_ID` / `CLIENT_SECRET`
- `IMO_CLIENT_ID` / `IMO_CLIENT_SECRET`
- `IMO_API_KEY` / `IMO_API_SECRET`

### Important remote-path note

Even when remote IMO is used, the module still needs local OMOP vocabulary files. IMO returns codes such as SNOMED or RxNorm, and the code then resolves those codes to OMOP `concept_id` values using your local `CONCEPT.csv`.

## Quick start

### 1. Run only the LLM stage

```bash
python cli.py llm \
  --schema-path ../protocol_schema_validator.json \
  --venice-timeout 1000 \
  --venice-model zai-org-glm-5:5 \
  --venice-model kimi-k2-5:4 \
  --enable-semantic-verification \
  --llm-verifier-model openai-gpt-54:4 \
  --llm-verifier-model claude-sonnet-4-6:4 \
  --pretty \
  "Characterize adults with type 2 diabetes treated with metformin in the last 5 years."
```

### 2. Run only the concept mapping stage

```bash
python cli.py concept-mapping \
  --schema-path ../protocol_schema_validator.json \
  --concept-mapping-source auto \
  --athena-concept-csv /data/athena/CONCEPT.csv \
  --athena-relationship-csv /data/athena/CONCEPT_RELATIONSHIP.csv \
  --athena-synonym-csv /data/athena/CONCEPT_SYNONYM.csv \
  --protocol-json ./protocol.json \
  --pretty
```

### 3. Run only the SQL stage

```bash
python cli.py sql \
  --schema-path ../protocol_schema_validator.json \
  --mapped-protocol-json ./mapped_protocol.json \
  --pretty
```

### 4. Run the full pipeline

```bash
python cli.py run \
  --schema-path ../protocol_schema_validator.json \
  --concept-mapping-source auto \
  --athena-concept-csv /data/athena/CONCEPT.csv \
  --athena-relationship-csv /data/athena/CONCEPT_RELATIONSHIP.csv \
  --athena-synonym-csv /data/athena/CONCEPT_SYNONYM.csv \
  --show-llm \
  --show-concept-mapping \
  --show-sql \
  --pretty \
  "What is the incidence of chronic kidney disease after type 2 diabetes diagnosis in adults aged 40 to 75?"
```

## CLI options that matter most

### Common options

- `--schema-path` — path to `protocol_schema_validator.json`
- `--pretty` — pretty-print JSON output
- `--save-output` — write final output to a file
- `--show-llm` — show the protocol generated by the LLM stage
- `--show-concept-mapping` — show the mapped protocol in a full `run`
- `--show-athena` — legacy alias for `--show-concept-mapping`
- `--show-sql` — show the SQL build result in a full `run`

### LLM options

- `--venice-model model[:retries]` — repeat to define fallback order
- `--venice-api-key` — overrides `VENICE_API_KEY`
- `--venice-base-url` — Venice endpoint base URL
- `--venice-timeout` — per-request timeout
- `--mock-protocol-json` — bypass Venice and inject a local protocol JSON file
- `--include-venice-system-prompt` — pass Venice-specific extra prompt parameters
- `--enable-semantic-verification` — run a second LLM pass that reviews the full protocol for semantic / clinical flaws
- `--llm-verifier-model model[:retries]` — verifier model order and retries
- `--llm-verifier-timeout` — timeout for the semantic verification step

### Concept mapping options

- `--concept-mapping-source auto|local|remote` — choose how the concept mapping stage uses its sources
  - `auto` — remote IMO first, then local lexical fallback if needed
  - `local` — local deterministic matcher only
  - `remote` — remote IMO only, with **no** lexical local fallback
- `--athena-api-base-url` — override the LOF base URL if needed
- `--athena-api-timeout` — timeout for the remote IMO/LOF path
- `--athena-concept-csv` — path to `CONCEPT.csv`
- `--athena-relationship-csv` — path to `CONCEPT_RELATIONSHIP.csv`
- `--athena-synonym-csv` — path to `CONCEPT_SYNONYM.csv`
- `--athena-candidate-limit` — max number of kept candidates
- `--athena-ambiguity-delta` — how close top scores can be before the result becomes ambiguous
- `--athena-minimum-match-score` — minimum score required for automatic mapping

Legacy compatibility flags:

- `--athena-disable-api` — legacy way to force local-only mode
- `--athena-prefer-local` — legacy way to force local-only mode

### Mapping-source behavior

#### `auto`

This is the default. The module:

1. tries remote IMO normalization through your LOF proxy,
2. extracts codes such as SNOMED or RxNorm from the IMO response,
3. resolves those codes to OMOP `concept_id` values using local `CONCEPT.csv`,
4. falls back to the deterministic local matcher if remote lookup fails or returns no resolvable OMOP concepts.

#### `local`

Use this when you want only the deterministic local matcher. Remote IMO is skipped entirely.

```bash
python cli.py concept-mapping \
  --schema-path ../protocol_schema_validator.json \
  --concept-mapping-source local \
  --athena-concept-csv /data/athena/CONCEPT.csv \
  --athena-relationship-csv /data/athena/CONCEPT_RELATIONSHIP.csv \
  --athena-synonym-csv /data/athena/CONCEPT_SYNONYM.csv \
  --protocol-json ./protocol.json \
  --pretty
```

#### `remote`

Use this when you want only the remote IMO path. The module still uses local `CONCEPT.csv` to resolve returned codes into OMOP `concept_id` values, but it will **not** fall back to the lexical local matcher if remote normalization fails or returns no resolvable codes.

```bash
python cli.py concept-mapping \
  --schema-path ../protocol_schema_validator.json \
  --concept-mapping-source remote \
  --athena-concept-csv /data/athena/CONCEPT.csv \
  --athena-relationship-csv /data/athena/CONCEPT_RELATIONSHIP.csv \
  --athena-synonym-csv /data/athena/CONCEPT_SYNONYM.csv \
  --protocol-json ./protocol.json \
  --pretty
```

If LOF/IMO credentials are missing in remote-only mode, the concept mapping stage raises an error instead of silently falling back.

## How the current LLM stage works

The current LLM layer is not asked to emit the full final protocol directly. Instead it:

1. asks Venice for a smaller **interpretation object**,
2. validates that interpretation object,
3. deterministically builds the full protocol JSON in Python,
4. optionally runs a second semantic verification pass using one or more stronger verifier models.

That design makes the protocol shape much more stable than asking the model to invent the entire final schema directly.

### Semantic verification

Use `--enable-semantic-verification` when you want a second LLM pass to review the protocol as a domain expert.

```bash
python cli.py llm \
  --schema-path ../protocol_schema_validator.json \
  --venice-model zai-org-glm-5:5 \
  --enable-semantic-verification \
  --llm-verifier-model openai-gpt-54:4 \
  --pretty \
  "Characterize adults with type 2 diabetes treated with metformin in the last 5 years."
```

The verifier receives:

- the original clinician question,
- the full current protocol,
- the full validator schema,
- instructions to keep schema validity while fixing semantic or study-design flaws.

If a reasoning-heavy verifier returns reasoning but no final JSON, the code automatically makes a follow-up call asking it to output only the corrected final protocol JSON.

## How the concept mapping stage works

The concept mapping stage processes `protocol["concept_sets"]`. For each concept set it uses:

- `raw_text`
- `domain`
- `standard_vocab_preference`

### Remote IMO path

The remote path uses your LOF service proxy:

1. request a bearer token from `/generate-access-token/`,
2. call `/imo/normalize`,
3. extract codes such as SNOMED or RxNorm from the response,
4. resolve those codes to OMOP concepts using the local vocabulary files.

### Local deterministic matcher

The local matcher is deliberately deterministic. It does **not** use embeddings or fuzzy LLM reasoning. It applies the same ordered rules every time:

1. normalized exact concept-name match
2. normalized exact synonym match
3. normalized prefix / substring match
4. token-overlap scoring
5. vocabulary and domain bonuses
6. standard-concept preference via `Maps to` resolution
7. ambiguity detection when top scores are too close

This stage also includes the parser hardening fixes that were added during debugging:

- delimiter auto-detection for ATHENA downloads that are really tab-delimited
- increased CSV field-size limit for very large files
- clearer file-versus-directory errors
- concept-code indexing so remote IMO codes can be resolved to OMOP concepts efficiently

### Mapping statuses

Each `concept_set` ends up with a `mapping` object whose status is one of:

- `mapped` — one acceptable concept was selected
- `ambiguous` — several close candidates exist and human review is needed
- `unmapped` — no candidate passed the thresholds
- `not_required` — supported by the schema, though the current pipeline normally uses `mapped`, `ambiguous`, or `unmapped`

### Protocol status after concept mapping

- all concepts mapped cleanly -> `executable`
- one or more ambiguous concepts -> `needs_review`
- one or more unmapped concepts -> `blocked`

## How to interpret each layer's output

### 1. LLM output

This is the structured protocol JSON produced from the clinician question. Key fields to inspect:

- `study_type`
- `target_cohort`
- `comparator`
- `outcome`
- `time_windows`
- `concept_sets`
- `protocol_status`
- `issues`

What success looks like:

- valid JSON according to `protocol_schema_validator.json`
- the right study type for the question
- meaningful cohort, comparator, outcome, and time-window definitions
- concept terms present in `concept_sets`

### 2. Concept mapping output

This is the same protocol JSON after mapping has been applied. Inspect:

- `concept_sets[*].mapping.status`
- `concept_sets[*].mapping.omop_concept_id`
- `concept_sets[*].mapping.candidate_concepts`
- `issues.warnings` for remote-versus-local behavior

Typical remote/local warning codes include:

- `IMO_REMOTE_MODE`
- `IMO_REMOTE_USED`
- `IMO_REMOTE_NO_MATCH`
- `IMO_API_FALLBACK`
- `IMO_REMOTE_FAILURE_DETAIL`
- `ATHENA_LOCAL_USED`
- `AMBIGUOUS_CONCEPT`

### 3. SQL output

This is the populated SQL plus SQL build metadata. The current SQL stage only accepts protocols whose concept mappings are already fully `mapped` and whose `protocol_status` is `executable`. It does **not** support the older reviewed/deblocked override behavior.

Inspect:

- `template_name`
- `parameters`
- `sql`

This layer answers: **what SQL would be run?** It does not execute anything.

## Recommended debugging workflow

### Step 1 — run only LLM

```bash
python cli.py llm ...
```

Check:

- correct study type
- correct concept extraction
- valid protocol JSON

### Step 2 — run only concept mapping

```bash
python cli.py concept-mapping ...
```

Check:

- mapping quality
- candidate concepts
- ambiguity or unmapped cases
- whether remote IMO or local matching was actually used

### Step 3 — run only SQL

```bash
python cli.py sql ...
```

Check:

- correct template selected
- correct concept IDs inserted
- readiness / executability assumptions

## Error handling philosophy

Each layer raises structured errors upward to the API and CLI. Examples:

- LLM errors:
  - Venice API failure
  - invalid model
  - timeout
  - interpretation validation failure
  - schema validation failure

- Concept mapping errors:
  - remote IMO auth or network failure
  - remote-only mode requested without credentials
  - malformed local vocabulary files
  - ambiguous or unmapped concepts

- SQL errors:
  - protocol not executable
  - missing mapped concept refs
  - unsupported event types or criteria

## Final note

This repository currently targets the **pre-execution** part of the RWE-Gen system:

- generate protocol
- validate protocol
- map concepts
- build SQL

It does **not**:

- execute SQL
- query an OMOP database
- compute final study results
